# Internal User Story Guidelines

## Purpose

This document defines the internal standards for writing user stories within the organization.

User stories describe business requirements from the perspective of the end user. Their purpose is to communicate business value, facilitate collaboration between stakeholders and developers, and provide enough information for implementation planning.

User stories are not technical specifications. They describe *what* the user needs and *why* the functionality is valuable, but they do not describe *how* the solution should be implemented.

All user stories generated by the system must follow these guidelines.

---

# Standard Structure

Every user story shall include the following information:

- Project
- User role
- Business goal
- Business reason
- Functional description
- Priority
- Story points
- Estimated effort

The information must be internally consistent.

The description should expand the business need without introducing implementation details.

---

# User Role

The role identifies the person or system that benefits from the requested functionality.

The role should represent a real actor involved in the business process.

Examples:

- Customer
- Administrator
- Sales Manager
- Human Resources Employee
- Support Technician
- Project Manager

Avoid using generic terms such as:

- User
- Person
- Client application
- System

Whenever possible, use the most specific role.

---

# Business Goal

The goal describes the functionality requested by the user.

A good goal answers the question:

"What does the user want to accomplish?"

Examples:

- create a new account
- manage users
- generate monthly reports
- upload documents
- review pending requests
- assign tasks

Goals should describe a single business capability.

Avoid combining multiple unrelated objectives in the same story.

---

# Business Reason

Every user story must explain the business value.

A valid reason answers the question:

"Why is this functionality useful?"

Examples:

- to improve productivity
- to reduce manual work
- to increase customer satisfaction
- to reduce processing time
- to improve data quality
- to simplify daily operations

Stories without a clear business value should be avoided.

---

# Functional Description

The description provides additional context.

It may explain:

- expected behavior
- business rules
- workflow
- restrictions
- important validations

The description should remain functional.

Do not include:

- programming languages
- database technologies
- API design
- framework names
- architectural decisions

Those details belong to technical tasks, not user stories.

---

# Priority

The organization uses four priority levels.

Low

Small business impact.

Medium

Normal priority.

High

Important functionality that should be implemented soon.

Blocking

Critical functionality that prevents business operations or blocks other work.

Priority should reflect business value rather than technical complexity.

---

# Story Points

Story points estimate relative implementation effort.

Allowed values are:

- 1
- 2
- 3
- 5
- 8

General interpretation:

1 → Very small change.

2 → Small feature.

3 → Medium feature.

5 → Large feature with moderate uncertainty.

8 → Large feature with significant uncertainty or complexity.

Story points measure effort, complexity and uncertainty together.

They are not equivalent to development hours.

---

# Estimated Effort

Estimated effort is expressed in hours.

The estimate provides an approximate implementation time.

Typical values range between:

- 2 hours
- 4 hours
- 8 hours
- 16 hours
- 24 hours

The estimate should remain realistic and proportional to the assigned story points.

---

# Writing Guidelines

A good user story should:

- describe one business need;
- provide measurable business value;
- be understandable by both business and technical teams;
- avoid implementation details;
- be implementable during a sprint;
- support later decomposition into technical tasks.

The story should remain concise while providing sufficient context.

---

# Common Mistakes

Do not create stories that describe technical work.

The following are NOT user stories:

- Create a database table.
- Implement a REST endpoint.
- Refactor authentication code.
- Configure Docker.
- Deploy to Azure.
- Create SQL indexes.

These activities represent implementation tasks.

The user story should instead describe the business functionality that requires those tasks.

---

# Expected Quality

A high-quality user story:

- focuses on business value;
- is clear and unambiguous;
- can be estimated consistently;
- enables technical planning;
- facilitates acceptance criteria definition;
- serves as the starting point for task generation.
