# INVEST Principles for High-Quality User Stories

## Purpose

The INVEST framework is the organization's standard for evaluating the quality of user stories before they are accepted into a sprint.

Every generated user story should satisfy these principles whenever possible.

The objective is to ensure that stories are understandable, valuable, estimable and ready for implementation.

---

# Independent

A user story should be as independent as possible.

Independent stories minimize dependencies between development tasks and allow teams to prioritize work more efficiently.

A story should not rely on another unfinished story in order to provide business value.

If dependencies exist, they should be minimized.

---

# Negotiable

A user story is not a contract.

It represents a conversation between business stakeholders and the development team.

The exact implementation details should remain open until planning or refinement sessions.

Stories should describe objectives rather than technical solutions.

Avoid including implementation decisions inside the story.

---

# Valuable

Every story must provide measurable business value.

If the user receives no benefit from the requested functionality, the story should be reconsidered.

Business value may include:

- reducing manual work;
- improving productivity;
- increasing customer satisfaction;
- improving information quality;
- reducing operational costs;
- improving decision making.

Stories without clear value should not be implemented.

---

# Estimable

The development team should be able to estimate the story.

If estimation is impossible, the story is probably too large or lacks sufficient information.

Stories with excessive uncertainty should be divided into smaller stories.

Clear requirements improve estimation accuracy.

---

# Small

Stories should be small enough to be completed within a single sprint.

Large stories increase uncertainty and implementation risk.

If multiple independent features are described, they should be separated into different stories.

A story should focus on one primary business objective.

---

# Testable

Every story must be testable.

The team should be able to determine objectively whether the implementation satisfies the requested functionality.

Well-defined acceptance criteria make stories easier to validate.

Stories that cannot be verified should be refined before development begins.

---

# Applying INVEST

When generating a new story, verify that:

- the requested functionality represents a single objective;
- business value is explicitly described;
- unnecessary technical details are avoided;
- the scope is reasonable;
- implementation effort can be estimated;
- the expected result can be validated.

Stories that satisfy these conditions are significantly easier to implement successfully.

---

# Common Anti-Patterns

Avoid stories that:

- combine unrelated business processes;
- describe technical implementation;
- have no identifiable user;
- provide no business value;
- contain ambiguous objectives;
- are too large to complete during one sprint.

These anti-patterns increase development risk and reduce planning accuracy.

---

# Quality Checklist

Before accepting a generated user story, verify that:

✓ A specific user role exists.

✓ The business goal is clearly defined.

✓ The business reason explains the expected value.

✓ The description provides enough context.

✓ The story remains implementation-independent.

✓ The scope is appropriate for one sprint.

✓ The story can be estimated.

✓ The functionality can be validated through acceptance criteria.
